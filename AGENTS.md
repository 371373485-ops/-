# AGENTS.md

本文件是“小红书爆款内容诊断 Agent 系统”的长期开发规范。所有后续代码修改、接口设计、Agent 设计、测试和文档更新，都应遵守本文件约定。

## 一、项目目标

本项目是“小红书爆款内容诊断 Agent 系统”，面向内容创作者，用于在合规边界内辅助用户优化小红书笔记内容。

系统允许用户：

1. 手动输入笔记标题、正文、标签、类目、封面文案等内容。
2. 上传用户自有或已获得授权的爆款样本。
3. 基于用户输入和自有样本进行内容评分。
4. 分析爆款内容规律和当前笔记问题。
5. 通过多 Agent 工作流完成标题改写、正文重构、标签推荐、封面文案生成、评论区引导生成。
6. 对输出内容进行风险审查，避免违法、违规、虚假宣传、夸大承诺和违规引流。

系统定位是“内容诊断与创作辅助工具”，不是平台自动化工具，不连接、不控制、不采集任何小红书账号或平台数据。

## 二、诊断报告可信度与深度要求

1. 诊断报告必须采用证据型诊断，不能只输出泛化建议。
2. 诊断必须让用户感到系统真正读懂了原文：应围绕用户提交的标题、正文、标签、类目、封面文案和自有样本进行分析。
3. 必须输出 `top_3_core_blockers` 或等价结构，用于呈现当前内容最核心的 Top 3 阻碍，并按优先级排序。
4. 必须模拟目标读者视角，说明目标读者看到标题、开头、正文、结尾时可能产生的疑问、兴趣、信任变化和行动意愿。
5. 必须分析内容结构，至少覆盖：
   - 开头 3 秒是否快速交代读者、场景、痛点或收益。
   - 信息层次是否清晰，是否便于扫读、完读和收藏。
   - 信任建立是否充分，是否有真实背景、判断依据、限制条件或过程说明。
   - 细节证据是否充分，是否存在只有结论、缺少细节支撑的问题。
   - 情绪共鸣是否自然，是否能让目标读者感到“这说的是我”。
   - 行动引导是否合规、具体，并优先引导站内评论、收藏或自我判断。
6. 必须检查可信细节是否充分。真实细节包括但不限于：用户真实场景、选择过程、使用条件、观察到的变化、失败经验、限制条件、适用/不适用人群。
7. 不得替用户编造真实经历、数据、使用效果、身份背书、案例、用户反馈、平台表现或第三方评价。
8. 当缺少支撑诊断或改写的真实细节时，必须输出 `missing_user_inputs` 或等价结构，明确列出建议用户补充的信息，而不是自行补全。
9. 改写结果必须附带 `rewrite_explanations` 或等价结构，说明每处改写解决了什么问题、依据哪条证据、预期改善点击/完读/信任/互动/合规中的哪一项。
10. 诊断语气应专业、具体、直接，但不得羞辱用户，不得使用贬低、嘲讽、攻击或制造焦虑的表达。

## 三、证据型诊断报告输出标准

每条核心问题应尽量包含以下字段或等价信息：

1. `field`：对应字段，例如标题、正文开头、正文结构、标签、封面文案、评论引导、风险表达。
2. `evidence`：原文证据、短引用或忠实片段概括。证据必须来自用户输入或用户授权样本，不能编造。
3. `judgement`：问题判断，明确指出哪里影响内容表现或合规可信度。
4. `why_it_matters`：为什么重要，说明它影响点击、完读、信任、互动、收藏、转化判断或合规中的哪一项。
5. `severity`：严重程度，可使用 P0/P1/P2/P3、high/medium/low 或等价优先级。
6. `revision_principle`：修改原则，说明应该遵循的方向和边界。
7. `example_fix`：示例改法。示例必须基于用户已提供信息进行保守改写；如缺少真实细节，只能给模板或提示用户补充，不得编造事实。

证据型诊断报告还应满足：

1. 优先输出具体问题，而不是抽象评价。禁止只说“增强吸引力”“优化表达”“突出卖点”。
2. 对原文证据的引用应短而必要；涉及用户敏感内容时应概括，不应在日志或无必要位置暴露正文原文。
3. `top_3_core_blockers` 应覆盖最影响结果的三个问题，通常优先考虑合规风险、标题点击、开头留存、信任证据、结构清晰度和行动引导。
4. `missing_user_inputs` 应列出为了更可信改写所需的真实信息，例如真实使用场景、具体步骤、时间范围、观察结果、失败条件、适用边界、可公开的非敏感数据。
5. `rewrite_explanations` 必须与改写结果一一对应，避免只给成品不说明依据。
6. 如用户目标涉及医疗、金融、收益、功效、未成年人、抽奖、站外引流或极限词，应提高严重程度并优先输出合规提醒和保守改法。
7. 如果系统无法从原文中找到足够证据，应明确说明“当前证据不足”，并转为提问式、模板式或待补充式建议。

## 四、强制禁止事项

1. 禁止实现小红书爬虫。
2. 禁止绕过任何平台访问限制。
3. 禁止自动登录账号。
4. 禁止自动发布内容。
5. 禁止保存用户 Cookie、Token、API Key 到代码。
6. 禁止把密钥写入 README、测试文件、日志。
7. 禁止生成违法、违规、虚假宣传、夸大承诺内容。
8. 禁止做无关重构。
9. 禁止调用小红书平台接口或模拟平台私有接口。
10. 禁止批量采集、下载、解析或复制平台内容。
11. 禁止设计规避平台审核、风控、反爬或访问限制的功能。
12. 禁止在示例数据中包含真实账号凭证、真实 Cookie、真实 Token 或真实 API Key。

如需求与上述禁止事项冲突，必须拒绝该需求，并提供合规替代方案，例如“用户手动上传自有 CSV 样本”。

## 五、技术栈

- Frontend: React + Vite + Tailwind
- Backend: FastAPI
- Database: SQLite，后续可替换 PostgreSQL
- AI: OpenAI API，必须支持 mock 模式
- Testing: pytest

默认开发和演示优先使用 mock 模式，确保没有 API Key 时也能运行、测试和演示。

## 六、项目结构说明

```text
frontend/
  src/
    api/
    components/
    styles/

backend/
  app/
    api/
      routes/
    agents/
    prompts/
    schemas/
    services/
    models/
    core/
  tests/
```

- `frontend`：前端应用目录，使用 React + Vite + Tailwind，负责用户输入、结果展示和调用后端 API。
- `frontend/src/api`：前端 API 请求封装，集中管理后端接口地址和请求逻辑。
- `frontend/src/components`：前端展示组件，例如诊断表单、评分面板、风险审查面板、结果卡片。
- `frontend/src/styles`：前端样式入口和全局样式。
- `backend`：后端应用目录，使用 FastAPI，负责接口、业务逻辑、Agent 编排、数据存储和测试。
- `backend/app`：后端核心应用包。
- `backend/app/api/routes`：HTTP 路由层，只处理请求、响应和必要的参数校验，不承载复杂业务逻辑。
- `backend/app/agents`：Agent 类目录。每个 Agent 独立成类，例如标题 Agent、正文 Agent、标签 Agent、风险审查 Agent。
- `backend/app/services`：业务服务层，负责协调 Agent、数据库、AI 服务、mock 服务和诊断流程。
- `backend/app/prompts`：Prompt 模板目录。真实 OpenAI 调用相关提示词集中放在这里，方便审查和版本管理。
- `backend/app/schemas`：Pydantic 请求和响应结构，所有 API 返回都应使用结构化 JSON。
- `backend/app/models`：数据库模型目录，SQLite 阶段用于保存样本、诊断历史等结构。
- `backend/app/core`：配置、环境变量读取、安全相关工具等基础设施。
- `backend/tests`：pytest 测试目录，覆盖健康检查、诊断接口、样本上传、Agent 服务和风险审查。

当前阶段尚未实现的目录可在对应功能阶段创建，但创建后应遵守上述职责边界。

## 七、启动命令

后端启动：

```bash
cd backend
python -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

如当前环境没有 `python` 命令，可使用本机实际 Python 路径替代。

前端启动：

```bash
cd frontend
npm install
npm run dev
```

默认访问地址：

```text
http://localhost:5173
```

默认后端地址：

```text
http://localhost:8000
```

## 八、测试命令

后端测试：

```bash
cd backend
pytest
```

如果 `pytest` 不在 PATH 中，可使用：

```bash
cd backend
python -m pytest
```

测试应优先覆盖：

1. `/health` 健康检查。
2. 诊断接口请求和响应结构。
3. mock AI 模式。
4. 样本上传和字段校验。
5. 风险审查规则。
6. Agent 输出结构。

## 九、开发约定

1. 每个 Agent 独立成类。
2. 每个 API 返回结构化 JSON。
3. 新增功能要尽量配套测试。
4. 新增环境变量必须同步更新 `.env.example`。
5. 修改接口时必须同步更新 `README.md`。
6. 优先小步改动，不做大规模无关重构。
7. 修改后必须说明测试情况。
8. 后端路由层保持轻量，复杂逻辑放入 `services` 或 `agents`。
9. Pydantic schema 是前后端契约来源，新增或修改字段时要保持命名清晰、类型明确。
10. AI 调用必须通过可替换的服务层封装，不能在路由或 Agent 中散落直接调用。
11. 所有真实 AI 调用都必须保留 mock fallback，避免无 API Key 时系统不可演示。
12. Prompt 必须避免要求模型生成违法、违规、虚假宣传、夸大承诺、违规引流内容。
13. 风险审查应保守处理，发现不确定内容时优先提示人工复核。
14. 示例数据必须是虚构或安全样例，不使用真实用户隐私、真实账号信息或真实平台凭证。
15. 前端页面应以工具体验为主，不做无关营销页。
16. 修改依赖时应说明原因，并保持依赖数量克制。
17. 生成或修改日志逻辑时，不得输出用户密钥、Cookie、Token、API Key 或敏感正文原文。

## 十、安全边界

系统的数据来源只能是：

1. 用户手动输入的内容。
2. 用户上传的自有样本。
3. 用户确认已获得授权的样本。

系统不得：

1. 连接小红书账号。
2. 访问小红书私有接口。
3. 抓取小红书页面。
4. 自动发布或调度发布任何平台内容。
5. 采集用户 Cookie、Token、账号密码或平台会话信息。
6. 保存、打印、提交或上传任何密钥。

所有 AI 生成结果都应经过风险审查。涉及医疗、金融、功效承诺、收益承诺、未成年人、站外引流、抽奖诱导、极限词和绝对化表达时，必须采用更保守的输出策略。
